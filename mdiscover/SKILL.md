# /mdiscover

Run this before /mlaunch. Run this before you've decided what to build.

The research phase is most valuable when it's not attached to a specific product
or launch. /mdiscover is a standalone command so you can do discovery work early,
reuse it across multiple skills, and separate "what do people actually want" from
"what am I building."

Output: `marketing/discovery/{date}/` — read automatically by /mlaunch, /mcmo, /mbrand.

## Preamble

```bash
_DATE=$(date +%Y-%m-%d)
_OUT="marketing/discovery/$_DATE"
mkdir -p "$_OUT"
echo "Discovery output: $_OUT"

# Check browse binary
B=""
[ -x "$HOME/.claude/skills/gstack/browse/dist/browse" ] && B="$HOME/.claude/skills/gstack/browse/dist/browse"
[ -n "$B" ] && echo "BROWSE: ready" || echo "BROWSE: unavailable (WebSearch fallback)"

# Check for existing discovery
LATEST=$(ls -td marketing/discovery/*/ 2>/dev/null | head -1)
if [ -n "$LATEST" ]; then
  DAYS_AGO=$(( ( $(date +%s) - $(date -r "$LATEST" +%s 2>/dev/null || echo $(date +%s)) ) / 86400 ))
  echo "EXISTING DISCOVERY: $LATEST ($DAYS_AGO days old)"
  [ "$DAYS_AGO" -lt 7 ] && echo "FRESH (< 7 days) — pass --refresh to re-run" || echo "STALE (> 7 days) — recommend re-running"
fi
```

If existing discovery is < 7 days old and user did not pass `--refresh`, ask:

> Recent discovery found from {date} ({N} days ago). Use existing research or refresh?
> A) Use existing — skip to summary
> B) Refresh — re-run all searches

## Phase 1: Problem Space Definition

Before searching, extract what you know about the problem space:

1. **From inline arg:** `/mdiscover "problem space description"` — use this directly
2. **From README.md** (if in a repo): H1 + first paragraph
3. **Interactive fallback:** Ask "What problem space are you exploring? Describe the
   user and the pain in one sentence."

State clearly before searching:
- Problem space: [one sentence]
- Likely user: [one sentence]
- Closest competitors to check: [2-3 names, best guess]

## Phase 2: Research Searches

Run all searches. Use WebSearch primarily; use `$B goto` + `$B text` for richer
page scraping when available.

### 2a. Reddit — raw pain language

```
Search 1: site:reddit.com "[problem space]" "how do you" OR "wish there was" OR "I hate" OR frustration
Search 2: site:reddit.com "[problem space]" OR "[user type]" "I just want" OR "why can't" OR "is there a way"
```

Extract verbatim:
- Exact phrases describing the pain (not paraphrased)
- Workarounds they're cobbling together
- "I wish X would just..." statements
- Emotional language: frustration, embarrassment, wasted time, cost

### 2b. Hacker News — technical buyer language

```
Search: site:news.ycombinator.com "Ask HN" [problem space]
```

Extract verbatim:
- How technical users frame the problem vs. non-technical users
- Tools they've evaluated and rejected, and why
- Price sensitivity signals
- Solutions they've built themselves (reveals depth of pain)

### 2c. Competitor reviews — what breaks, what sticks

Identify 2-3 closest competitors. Search:
```
site:g2.com "[competitor]" reviews
site:trustpilot.com "[competitor]"
"[competitor]" reviews site:reddit.com
```

Extract verbatim from 1-star and 5-star reviews:
- 1-star: exact failure modes, exact frustrated phrases
- 5-star: what they'd be lost without (this is what you must match or beat)
- Phrases that appear in multiple reviews (resonant = real)

### 2d. Job postings — operational pain in employer language

```
Search: "[job title most likely to buy this]" job description [workflow related to problem]
```

Extract:
- How companies describe the problem in hiring language
- What tools/skills they require (reveals status quo)
- Outcomes they promise to the hire (reveals what success looks like to the buyer)

### 2e. Competitor positioning — what's been said to death

Visit 1-2 competitor homepages (use `$B goto` + `$B text` if available, else search):
```
Search: "[competitor] homepage" OR "[competitor] pricing page"
```

Extract verbatim:
- Their exact headline and subheadline
- Their top 3 claimed benefits
- Their CTA text

**Record this carefully.** What every competitor says is what you do NOT say.

## Phase 3: Synthesis

Write `marketing/discovery/{date}/RESEARCH.md`:

```markdown
# Discovery Brief — [Problem Space] — [Date]
Generated by /mdiscover. Read by /mlaunch, /mcmo, /mbrand.

## Problem space
[One sentence]

## Likely user
[One sentence — best guess from research]

## Real user language (direct quotes)
Source each quote.
- "[exact quote]" — r/[subreddit] / HN / G2 / Trustpilot
...

## Current workarounds
- [specific workaround with source]
...

## Competitor positioning — what's been said to death
| Competitor | Headline | Top claims |
|-----------|----------|------------|
| [name] | "[exact headline]" | [claim 1], [claim 2] |
...

## Competitor failure modes (from 1-star reviews)
- [Competitor]: "[exact quote]" — [source]
...

## The gap
[1-2 sentences: what every competitor says vs. what nobody is saying yet]

## Resonant phrases (use verbatim or close)
Phrases that appeared in multiple independent sources:
- "[phrase]" — appeared in: [source 1], [source 2]
...

## Job posting signals
- Required tools: [list — reveals status quo]
- Promised outcomes: [list — reveals what success looks like]

## Who hurts most
[Which user segment expressed the most acute, specific pain? Quote the evidence.]

## Search quality notes
[Any searches that returned poor results, paywalled, or skipped — so downstream
skills know what's missing from this brief]
```

## Phase 4: Staleness Marker

```bash
echo "{\"date\":\"$_DATE\",\"problem_space\":\"[PROBLEM_SPACE]\",\"searches_run\":5,\"quotes_found\":N}" \
  > "$_OUT/.meta.json"
echo "Discovery complete: $_OUT/RESEARCH.md"
echo "This brief is read automatically by /mlaunch, /mcmo, and /mbrand."
echo "Re-run with /mdiscover --refresh if your problem space shifts."
```

## Completion

Report: DONE — RESEARCH.md written, meta file created.
Report: DONE_WITH_CONCERNS — if any search returned < 3 usable quotes (note which).
Report: BLOCKED — if no searches returned results and no fallback context available.
